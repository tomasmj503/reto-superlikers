# A2 — Sondeo de endpoints (hallazgos empíricos)

Ejecutado contra `https://api.superlikerslabs.com/v1`, campaña `3z`, entorno labs, vía n8n MCP (workflow diagnóstico `9N5BNAWdVJLlbgyX`).

## IP whitelist: no aplica en labs
Labs no bloquea por IP. Las llamadas desde el droplet reciben respuestas de negocio (4xx), nunca bloqueo de red. Se construye sin preocuparse por whitelist en el entorno de pruebas.

## Credencial
Header `Authorization: Bearer <api_key>` + campo `api_key` en el body de cada llamada. El valor vive en `$env.API_KEY` del contenedor Docker.

## Campo celular: es `celular`, no `cellphone`
Confirmado vía `POST /forms/signup/info` (endpoint de configuración del formulario de campaña). El campo se llama `celular` en `properties`. Se almacena con prefijo de país: se envía `3001234567` y la API lo devuelve como `+573001234567`.

## `GET /participants/search`: solo resuelve por `email`
El brief sugería buscar por celular. Empíricamente, cualquier búsqueda por celular (en 3 formatos) devuelve `404 code_error 20`. Solo el campo `email` produce match. El `distinct_id_field_name` del formulario es `email`.

**Formato correcto del query** (objeto anidado, no string):
```
query: { email: "correo@ejemplo.com" }
```
n8n lo serializa como `query[email]=…`. Mandar `query` como string plano → `422 code_error 18 "parametro query invalido"`. Ese era el bug del sondeo inicial.

## `POST /participants`: campos reales del formulario `3z`
El form tiene 11 campos. Los requeridos para el flujo:

| Campo | Tipo | Flags |
|-------|------|-------|
| `nombre-completo` | text | opcional |
| `cedula` | text | requerido + único |
| `email` | user_email | requerido + único (= `distinct_id`) |
| `password` | password | requerido |
| `celular` | phone | requerido + único |
| `ocupacion` | select | requerido |
| `tags` | check_box_group | requerido |

**Nota:** el campo de nombre es `nombre-completo` (con guion), no `first_name`/`last_name`.

## Valores válidos de `ocupacion` y `tags`

`ocupacion` (enviar el `value`):
- `ocupacion_1` → "Ocupación 1"
- `ocupacion_2` → "Ocupación 2"
- `ocupacion_3` → "Ocupación 3"
- `ocupacion_4` → "Ocupación 4"

`tags` (array de `value`):
- `admin`, `grupo_1`, `grupo_2`, `grupo_3`

## Impacto en el flujo
El paso 2 original ("buscar por celular") no es viable. El flujo debe pedir el correo temprano y buscar por `email`. El celular llega del `from` de WhatsApp y se usa como llave de `conversation_state`, normalizado a 10 dígitos.
