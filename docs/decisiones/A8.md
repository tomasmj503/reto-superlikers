## Bugs cazados en E2E (oro para el README)

### Bug 1 — Pérdida de correo en la rama de registro

**Síntoma:** `POST /participants` salía con `email` vacío → `422 "email no puede estar en blanco"`.

**Causa raíz:** al procesar el correo (`esperando_correo`), `Decidir` ponía el correo en su `patch`. Pero la acción era `buscar`, y `Procesar Búsqueda` construía su **propio** `patch` para la rama "no encontrado" (`{ paso: 'esperando_nombre', participante_existe: false }`) **sin incluir el correo** — pisando el patch del `Decidir`. Resultado: el correo se persiste durante la búsqueda pero no en el estado de "no encontrado, ir a registro".

**Fix (A8):** en `Procesar Búsqueda`, rama `else` (no encontrado por email): incluir `correo` y `distinct_id` en el patch, extrayéndolos de `d.search_query.email`.

```javascript
// FIX A8: arrastrar el correo al patch
const correoBuscado = (d.search_query && d.search_query.email) || '';
out.patch = { paso: 'esperando_nombre', participante_existe: false,
              correo: correoBuscado, distinct_id: correoBuscado };
```

**Por qué no se detectó antes:** las pruebas de A5/A6 usaban `qatest@jutilabs.com` (usuario existente), que nunca tocaba la rama de registro. La E2E de usuario nuevo lo expuso.

### Bug 2 — Falso positivo en detección de duplicado

**Síntoma:** `POST /participants` con email vacío devolvía `422 "datos no validos para crear el participante, code_error 21"`. El nodo `Procesar Registro` lo interpretaba como "duplicado = éxito" y respondía "¡Listo, quedaste registrado!" cuando en realidad no se creó nada.

**Causa raíz:** la condición `dup` buscaba `code_error 21` en el blob de la respuesta, que aparece tanto en duplicados reales (`"Ya esta siendo usado"`) como en errores de validación (`"no puede estar en blanco"`).

**Fix A8 (primera versión):** limitar `dup` a solo `"ya esta siendo usado"` e introducir guard `!invalid`:

```javascript
const dup = blob.indexOf('ya esta siendo usado') !== -1;
const invalid = blob.indexOf('no validos') !== -1 || blob.indexOf('no puede estar en blanco') !== -1;
if ((created || dup) && !invalid) { /* éxito */ }
```

**Bug secundario en el fix (A8b):** `"datos no validos"` aparece también cuando el celular está duplicado (`"celular: Ya esta siendo usado"`). El guard `invalid` era demasiado amplio y rechazaba casos que son idempotentes.

**Fix A8b (definitivo):** el guard `invalid` solo dispara cuando NO hay `"ya esta siendo usado"`:

```javascript
const dup = blob.indexOf('ya esta siendo usado') !== -1;
// Solo "campo en blanco" es error real. "Ya esta siendo usado" (cualquier campo único)
// es idempotente — el participante existe, avanzar a foto.
const invalid = !dup && blob.indexOf('no puede estar en blanco') !== -1;
if ((created || dup) && !invalid) { /* éxito */ }
```

## Externalización de variables de entorno (A8)

`CAMPAIGN` y `BASE_URL` estaban hardcodeadas en 5 nodos. El checklist de entrega las exige como env vars. Se externalizaron con **fallback** para que sea un cambio puramente aditivo:

```javascript
// En Code nodes:
const BASE = ($env.BASE_URL || 'https://api.superlikerslabs.com/v1');
campaign: ($env.CAMPAIGN || '3z')

// En HTTP Request nodes (URL):
={{ ($env.BASE_URL || 'https://api.superlikerslabs.com/v1') }}/participants

// En parámetros de form:
={{ $env.CAMPAIGN || '3z' }}
```

Nodos afectados: `Decidir`, `SL Buscar`, `A6 · Comprar + Aceptar`, `SL Registrar`, `A6 · Subir Foto`.

Si `$env.CAMPAIGN` o `$env.BASE_URL` no están declaradas, el flujo usa los valores por defecto y sigue funcionando. Cero regresión.
