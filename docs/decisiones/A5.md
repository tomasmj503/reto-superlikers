# A5 — State Machine

Workflow: `JUTILABS · Reto SL · Flujo Completo` (ID `hMUrIGCkq3qmUrA2`). Construido sobre el plumbing verificado de A4.

## Arquitectura del flujo

**Tabla de transiciones en un solo nodo `Decidir`.** El `Switch` (`Router Acción`) enruta por tipo de acción (`responder` / `buscar` / `registrar` / `foto`), no por cada uno de los 10+ estados. Menos nodos, menos superficie de error, una sola fuente de verdad para la lógica de transición.

**Copy separado de la lógica.** Todos los textos del bot viven en `prompts/copy.json` y se cargan desde un nodo `Cargar Config` al inicio del flujo. El nodo `Decidir` no tiene ningún string hardcodeado.

**Identificación celular → email con fallback.** El celular llega solo en el `from` de WhatsApp. Se intenta `search` por `cellphone` primero (honra el brief); como `3z` solo resuelve por `email`, cae a pedirle el correo al usuario. Resiliente a ambas configuraciones del API.

**Persistencia upsert.** Nodo `IF ¿Existía?` → rama `update` / rama `insert`. Se usa `autoMapInputData` para evitar el bug de columnas vacías del mapeo manual de Supabase.

**Idempotencia.** Un registro duplicado (`422 celular/cedula/email Ya esta siendo usado`) se trata como éxito y el flujo avanza. Evita bloquear a usuarios que retoman una conversación interrumpida.

**Convergencia limpia.** Las 4 ramas confluyen en un nodo `NoOp` (`Normalizar Salida`) antes de `Enviar WhatsApp` y la persistencia.

## Bugs cazados

### 1. Check constraint de `celular`: 10 dígitos exactos
El campo `celular` en `conversation_state` tiene constraint `^[0-9]{10}$`. El `from` de WhatsApp trae 12 caracteres (con prefijo `57`). Fix: normalizar el `from` a 10 dígitos (`from.replace(/^57/, '')`) antes de usarlo como llave de `conversation_state`. El formato `+57XXXXXXXXXX` se usa solo para enviar mensajes por WhatsApp.

### 2. Enum de `paso` desalineado con la state machine real
A3 dejó un enum provisional que no incluía los 15 estados finales de A5. Se aplicó la migración `a5_align_paso_check_to_final_state_machine` (drop del constraint + add con los 15 estados correctos). Sin esta migración, cualquier insert con un estado no contemplado falla silenciosamente o lanza error de constraint.

### 3. Método de `search` de A2
Construir la URL de búsqueda a mano en el nodo HTTP Request fallaba. La solución fue replicar el método exacto validado en A2: nodo Code `SL Buscar` usando `helpers.httpRequest` con el objeto `qs: { campaign, api_key, query: { email } }`. n8n serializa esto como `query[email]=…`. Mandar `query` como string JSON plano → `422 code_error 18`.

### 4. La respuesta de `/participants/search` viene envuelta en `body.object`
El participante no está en `body.participant` sino en `body.object`. Era la causa de que el flujo reportara "no encontré cuenta" con un participante que sí existía en labs. Confirmado ejecutando el workflow diagnóstico A2 en vivo durante el desarrollo de A5.

### 5. `.first()` vs `.item` en nodos Code dentro de ramas
Dentro de ramas del Switch, usar `.first()` en un nodo Code rompe el item pairing. La solución es usar `.item` directamente.

## Estructura de prompts (repo)

- `prompts/copy.json` — fuente de verdad del copy del bot (todos los textos por paso)
- `prompts/factura_system.txt` — system prompt para la IA de lectura de factura (A6)
- `prompts/README.md` — instrucciones de uso y convención de llaves

El nodo `Cargar Config` en n8n es el reflejo desplegado de `copy.json`. n8n self-hosted no importa archivos en runtime, por lo que el JSON se copia manualmente al nodo al actualizar el copy.
