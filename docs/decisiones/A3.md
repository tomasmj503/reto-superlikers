# A3 — Schema Supabase

Proyecto `superlikers` (ref `kphactyefgkdlwpyqfzi`, región `us-east-1`). Cuenta Supabase nueva e independiente, creada exclusivamente para el reto.

## Tabla `transactions` (log permanente, append-only)

Campos exactos que exige el reto: `celular`, `correo`, `ref_factura`, `puntos`, `estado`, `"timestamp"`.

Campos extra añadidos: `id_actividad_foto` (para trazabilidad del `POST /photos`), `csat` (micro-encuesta 1–5), `error_type` (para escalation rate del dashboard). PK surrogate `id` uuid.

**Nota:** el campo se llama `"timestamp"` (con comillas dobles en SQL) porque ese nombre colisiona con el tipo Postgres. En n8n y en el dashboard siempre referenciarlo con comillas dobles.

## Tabla `conversation_state` (working memory del bot)

Una fila por `celular`, se resetea por flujo completo. Campos de estado: `paso` (enum, 15 valores), `distinct_id`, `nombre`, `correo`, `cedula`, `ocupacion`, `tags` (array), `participante_existe`, `id_actividad_foto`, `datos` (jsonb libre), `intentos`, `last_activity_at`, timestamps de creación/actualización.

Trigger `updated_at` para mantener `last_activity_at` actualizado automáticamente.

## Decisión de idempotencia a nivel DB
Índice único parcial `uq_transactions_ref_completada` sobre `(ref_factura) WHERE estado='completada'`. Permite reintentos fallidos (no se bloquean) pero hace físicamente imposible otorgar puntos dos veces por la misma factura. Es la segunda línea de defensa, complementaria al chequeo en la aplicación.

## RLS activo desde el día 1
Verificado `true` en creación. Cuatro policies explícitas:
- `service_role` → ALL (n8n usa este rol)
- `authenticated` → SELECT (dashboard Next.js)
- `anon` → denegado por defecto

`service_role` tiene `BYPASSRLS`; las policies documentan la intención y blindan si el bypass se revoca en el futuro.
