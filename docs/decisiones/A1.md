# A1 — Plan y decisiones técnicas iniciales

## Orquestación: n8n (self-hosted en DigitalOcean)
n8n fue la opción recomendada en el brief y la más adecuada para el flujo: permite combinar nodos HTTP, lógica condicional, ejecución de código JS y webhooks sin levantar un backend custom. Self-hosted en DigitalOcean para tener control total sobre variables de entorno y credenciales.

## Canal: WhatsApp Business Cloud API (Meta)
Canal nativo del reto. La API Cloud de Meta elimina la necesidad de un BSP intermedio y permite recibir mensajes multimedia (fotos de tickets) directamente.

## Modelo IA para lectura de factura: OpenAI con Structured Outputs (`json_schema` estricto)
El brief sugería Claude u OpenAI. Se eligió OpenAI porque su modo `json_schema` garantiza el JSON a nivel de API: si el esquema no se cumple, la API reintenta internamente. Elimina la clase de error "el modelo devolvió algo que no es JSON". La imagen se pasa directa al modelo (sin OCR intermedio).

## Validación determinista post-extracción
El modelo extrae; el código valida. Después de recibir el JSON de la IA, un nodo Code verifica: `legible === true`, `ref` no vacío, `products` no vacío, `price` y `quantity` numéricos. Cualquier falla → pedir nueva foto.

## Estado conversacional y log: Supabase
Dos tablas: `conversation_state` (working memory del bot, una fila por celular) y `transactions` (log append-only permanente). RLS habilitado desde el día 1, no retroactivamente.

## Dashboard: Next.js + Supabase, deploy en Vercel
Métricas reales contra el log de `transactions`. Sin números inventados: lo que no tiene datos se declara como "listo para medir cuando haya datos históricos".

## Seguridad (no negociable)
La `api_key` vive en `$env.API_KEY` del contenedor Docker de n8n. Nunca hardcodeada en un nodo, nunca en el repo, nunca en Notion. El export del `.json` se revisa antes de commitear.
